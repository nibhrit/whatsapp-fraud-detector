Step 1 — Connecting to Claude

How does our code talk to Claude?

We use Anthropic's Python library. Think of it like a phone call — we dial up Claude, give it instructions and a question, and it talks back.

import anthropic

# Create the "phone" — one client, reused for every message
client = anthropic.Anthropic(api_key="sk-ant-...")

# Make a call to Claude Haiku
response = client.messages.create(
  model="claude-haiku-4-5-20251001",
  max_tokens=1024,
  system="You are a fraud expert. Respond only in JSON.",
  messages=[{"role": "user", "content": "Is this fraud? ..."}]
)
system vs messages: The system prompt is Claude's permanent instructions — it defines its role and rules. The messages list is the actual conversation. Think of system as the job description, messages as the work itself.
Step 2 — Getting structured output

Why JSON? How do we force Claude to return it?

1. Tell Claude in the system prompt: "Respond ONLY with a valid JSON object — no markdown, no text outside the JSON."

2. Parse the response text in Python:

raw = response.content[0].text.strip()

# Claude sometimes wraps JSON in ```json ... ``` — strip that
if raw.startswith("```"):
  raw = raw.split("\n", 1)[1].rsplit("```", 1)[0]

result = json.loads(raw)
# result = { "verdict": "FRAUD", "confidence": 95, "explanation": "..." }

How does the JSON become the card in the WhatsApp UI?

async function analyse(id, btn) {

The card colour (red vs green) is just CSS. The JS checks data.verdict === "FRAUD" and applies the .fraud or .legit class — which have different background and border colours defined in the stylesheet.
The core idea

What is a vector?

A vector is just a list of numbers. The trick is that these numbers are calculated in a way that captures meaning — not just keywords.

An embedding model (we use all-MiniLM-L6-v2) reads a sentence and converts it into ~384 numbers. Sentences with similar meanings get similar numbers.

Simple analogy: Imagine every sentence is a point on a map. "You've won a prize" and "Claim your reward now" are plotted close together. "What time is dinner?" is plotted far away. Vectors are the GPS coordinates of meaning.

What ChromaDB does

ChromaDB is a filing cabinet for vectors

We have 21 fraud patterns. ChromaDB stores each one as a vector. When a new message arrives:

1. Embed the message — convert it to a vector using the same model that embedded the patterns
2. Search — find the 3 pattern vectors that are closest to the message vector
3. Return — hand back those 3 patterns as text, with their distance scores
results = collection.query(
  query_texts=["aapka KYC expire ho raha hai"],
  n_results=3
)

# returns:
# 1. KYC Expiry Threat distance: 0.88 ← winner
# 2. Account Suspended distance: 1.37
# 3. Card Blocked Alert distance: 1.41
Key insight: ChromaDB doesn't do keyword search. It does semantic search. That's why a Hinglish sentence matches an English-language pattern — the meaning is the same even though the words are different.
The concept

What is RAG?

RAG = Retrieval Augmented Generation. A technique for giving an LLM specific, up-to-date knowledge at request time — without retraining it.

Without RAG

You ask Claude: "Is this fraud?"

Claude uses only its training data — broad, generic knowledge. It doesn't know your curated Indian fraud patterns.
With RAG

You ask: "Given these 3 specific patterns… is this fraud?"

Claude reasons against your library. More accurate, more specific, more controllable.
The three steps

Retrieve → Augment → Generate

R — Retrieve: Search ChromaDB for the 3 patterns most similar to the incoming message.
A — Augment: Inject those 3 patterns into the prompt as context. The prompt is now augmented with retrieved knowledge.
G — Generate: Claude reads the augmented prompt and generates a verdict, reasoning against those specific patterns.
The magic bit

How does the model "adapt instantly"?

Traditional ML: Add a new fraud pattern → retrain the model → redeploy → hours or days
RAG approach: Add a new pattern to fraud_patterns.py → run ingest_patterns() → it's in ChromaDB → next matching message retrieves it → Claude sees it → done in seconds

The model (Claude) never changes. Only the retrieval library changes. The LLM is like a brilliant analyst — you hand it different reference documents each time.

Real-world implication: When a new scam variant appears in India, a fraud analyst adds it to the pattern library. Zero ML engineering required. The system is immediately smarter.
FastAPI

What is FastAPI?

FastAPI is a Python library for building APIs — it turns a Python function into something that can be called with a URL over the internet.

# Without FastAPI: just a Python function
def analyse(message):
  return { "verdict": "FRAUD" }

# With FastAPI: that same function is now a URL endpoint
@app.post("/analyse")
def analyse_message(request: AnalyseRequest):
  return analyse(request.message)

# Anyone can now call:
# POST https://your-app.onrender.com/analyse
# { "message": "Your KYC is expiring..." }
Pydantic models define the exact shape of requests and responses. FastAPI validates every request automatically — if a field is missing or the wrong type, it returns a clear error without your code ever running.
CORS

What is CORS and why do we need it?

CORS = Cross-Origin Resource Sharing. A browser security rule that blocks JavaScript from calling a different domain than the page it's on.

The problem: Our frontend is on nibhritm.github.io. Our API is on onrender.com. Two different domains = blocked by default.
The fix: The backend tells the browser "I accept cross-origin requests" by sending special HTTP headers.
app.add_middleware(
  CORSMiddleware,
  allow_origins=["*"], # accept from any domain
  allow_methods=["GET", "POST"],
  allow_headers=["*"],
)

allow_origins=["*"] means "accept from everywhere" — fine for a portfolio demo. In production, restrict this to your exact frontend domain for security.

Startup

What is the lifespan function?

We need ChromaDB populated before any requests arrive. The lifespan function runs once when the server starts — like a setup routine before the shop opens.

@asynccontextmanager
async def lifespan(app):
  ingest_patterns() # runs once at startup
  yield # server now handles requests

app = FastAPI(lifespan=lifespan)
The yield is the dividing line. Before it: startup. After it: shutdown cleanup. This is how FastAPI knows what to run at the beginning and end of the server's life.
End-to-end

What happens when you tap 🛡 Analyse?

1
Frontend — Browser

User taps 🛡 Analyse on a message

JavaScript reads the text of the bubble the user clicked. It calls fetch(API_URL + "/analyse", { method: "POST", body: JSON.stringify({ message }) }). The browser sends this HTTP request to the Render server. While waiting, it shows a loading spinner.

↓
2
FastAPI — Render server

POST /analyse receives the request

FastAPI parses the JSON body into an AnalyseRequest Pydantic object and validates it (is message a non-empty string?). Then calls analyse(request.message) from rag_pipeline.py.

↓
3
ChromaDB — Vector DB

Retrieve the 3 most relevant fraud patterns

The message is embedded into a vector using all-MiniLM-L6-v2. ChromaDB searches its 21 stored pattern vectors for the 3 closest matches by distance. Returns the pattern name, category, description text, and distance score for each.

↓
4
RAG — Prompt construction

Build the augmented prompt

The 3 retrieved patterns are formatted into a readable block and injected into the user message: "RETRIEVED FRAUD PATTERNS: [pattern 1] [pattern 2] [pattern 3] --- MESSAGE TO ANALYSE: [original message]". The system prompt stays constant across all requests.

↓
5
Claude Haiku — Anthropic API

Claude generates the verdict

Claude sees: its role instructions, the 3 most relevant fraud patterns as context, and the message. It detects the script/language, reasons against the specific patterns, and returns a JSON object: verdict, confidence, pattern name, explanation, recommendation, language. Takes ~1-2 seconds.

↓
6
FastAPI — Response

Parse, validate, return to browser

The raw text from Claude is stripped of markdown fences if present, then parsed with json.loads(). FastAPI serialises it back to JSON and sends it as the HTTP response with a 200 status code.

↓
7
Frontend — Browser

Render the AI Shield card

JavaScript receives the JSON. renderResult() builds an HTML card using template literals, applying .fraud or .legit CSS class based on data.verdict. The card slides into the DOM directly below the analysed message bubble. Total round-trip: ~2-3 seconds.

Every component has one job: Browser JS → FastAPI → ChromaDB → Claude API → FastAPI → Browser JS. Built in layers, tested layer by layer.
